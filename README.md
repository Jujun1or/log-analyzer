# Log Analyzer — многопоточный анализатор NGINX-логов

## Обзор

Многопоточное CLI-приложение для обработки, фильтрации и анализа логов NGINX:

- Поддерживает **локальные файлы**, **URL-ресурсы**, **glob-шаблоны**
- Считывает данные в **батчах**, используя виртуальные потоки (Project Loom)
- Консьюмеры и продьюсеры работают через **BlockingQueue**
- Обеспечивает **fail-fast** поведение и детальное логирование
- Генерирует отчёты в форматах:
  - **JSON**
  - **Markdown**
  - **AsciiDoc**

Приложение оптимизировано для **скорости, многопоточности и низкого потребления памяти**.

## Возможности

### Источники логов

| Тип              | Пример                                      | Особенности |
|------------------|----------------------------------------------|-------------|
| Локальный файл   | `access.log`                                | Проверяется существование и расширение |

## Архитектура

### CLI-уровень

- **Application** — точка входа, вызов валидатора и оркестратора
- **Arguments**, **ArgumentsValidator** — DTO аргументов и fail-fast проверки

### Разбор источников
- **LogSourceResolver** — нормализация путей, раскрытие glob, проверка URL

### Парсинг логов
- **LineParser (NginxLineParser)** — разбор строки NGINX-лога  
  Логирует некорректные строки с уровнем **WARN**.

### Многопоточный конвейер

#### Продюсеры
- **LocalBatchReader**, **RemoteBatchReader**
- Читают файлы/HTTP-стрим **батчами**, прогоняют через time-filter
- Работают в **виртуальных потоках**

#### Консьюмеры
- **BatchProcessor** — расчёт статистики по батчу
- **GlobalStatsAggregator** — потокобезопасное слияние статистик

#### Pipeline
- Очередь `BlockingQueue<Batch>`
- Пул продьюсеров (`Executors.newVirtualThreadPerTaskExecutor()`)
- Пул консьюмеров
- Механизм `POISON_PILL` для корректного завершения потоков
- Полная обработка `IOException` + fail-fast/exit-codes

### Формирование отчёта
- **ReportFormatter** (интерфейс)
  - **JsonFormatter**
  - **MarkdownFormatter**
  - **AdocFormatter**

## Производительность

 **Многопоточность на основе Project Loom (virtual threads)**  
Каждый файл обрабатывается своим виртуальным потоком → масштабируется на десятки тысяч файлов.

 **Конвейер “reader → queue → consumer”**  
Продюсеры не блокируют консьюмеров и наоборот.

 **Память используется эффективно**  
- Жёсткие размеры batch’ей  
- Нет удержания больших коллекций строк  
- Обработка ведётся потоково (streaming)

 **I/O оптимизировано**  
- Локальные файлы читаются `BufferedReader`  
- Удалённые — через `HttpClient` + streaming API  
- Никакого чтения “весь файл в память”
